# 產品藍圖

[English](ROADMAP.md) | 繁體中文

本藍圖反映目前的產品方向，已納入章節工作流、審查歷史、修稿 diff、專案設定、Docker 支援與本地備份 / 還原等功能。

## 產品願景

一款本地 AI 寫作工作室，讓任何人——不需要任何技術背景——只要安裝好工具，就能建立自己的故事世界、生成完整草稿、審查、一步一步修成自己喜歡的模樣。不需要雲端、不需要訂閱、不需要寫程式。

預期的使用者旅程：

```
安裝 → 建立故事世界 → 生成草稿 → 審查 → 修稿 → 匯出
```

## 現階段重心

Novel Assistant 目前最強的定位是本地優先的小說審查工作站。

下一步不是繼續疊加孤立功能，而是深化三個產品層次：

- 稿件結構與寫作流程
- 可控制的故事專屬 RAG 行為
- 更安全的長期專案管理

## Priority 0：首次體驗

目標：確保任何使用者只要 clone repo 並執行 `docker-compose up`，就能在幾分鐘內得到可用的工具，不需要任何手動步驟。

- 首次啟動時自動 pull 所需 Ollama 模型（`llama3.2`、`nomic-embed-text`）
- 加入 Ollama 健康檢查，讓 app 等模型伺服器就緒後再接受請求
- 在 `data/chapters/` 加入範例章節，讓審查頁面有內容可以立即試用
- 修正 Docker 快速開始文件，將 `cp .env.example .env` 列為第一步

### 依硬體規格推薦模型

使用者不應該需要知道模型名稱。設定頁應根據使用者的硬體推薦合適的模型。

- 在設定頁加入硬體規格選項：
  - 入門（4GB VRAM）→ `llama3.2:3b`——速度快，基本品質
  - 標準（8GB VRAM）→ `llama3.1:8b`——平衡（推薦預設）
  - 進階（24GB VRAM）→ `gemma3:27b`——繁體中文強，品質高
  - 旗艦（48GB+ VRAM）→ `llama3.3:70b`——接近雲端服務品質
  - 自訂 → 可輸入任意 Ollama 模型名稱
- 選擇規格後自動更新 `LLM_MODEL`，並在背景觸發 `ollama pull`
- 嵌入模型（`nomic-embed-text`）固定不對使用者開放——它在各規格下表現都很好

為什麼這很重要：

- 前五分鐘的體驗決定開源使用者會繼續用還是直接關掉
- 相關工作追蹤於 Issues #20–24，是後續成長的前提
- 使用者不應該需要知道什麼是量化模型才能得到好的輸出

## Priority 1：引導式故事世界設定

目標：讓使用者透過引導式問答介面建立角色設定、世界觀筆記與風格偏好，不需要任何 Markdown 知識。

目前設定一個專案需要手動編寫格式化的 `.md` 檔案，並填入特定的中文或英文欄位標頭。這對非技術使用者來說是一道隱形的牆。

- 加入設定精靈頁面，逐步引導使用者建立故事世界：
  - 故事前提與類型
  - 角色（姓名、個性、核心恐懼、說話風格等）透過表單欄位填寫
  - 世界觀（背景、規則、陣營、地點）透過自由輸入提示填寫
  - 寫作風格偏好（語氣、節奏、視角、禁忌句型）
- 系統自動生成格式正確的 `data/characters/`、`data/worldbuilding/` 與 `data/style/` 檔案
- 使用者可以在之後編輯這些檔案以進行更深入的調整，但永遠不需要碰它們才能開始使用
- 加入「編輯故事世界」頁面，讓使用者不必接觸原始檔案就能更新設定

### 風格指南範例段落

每位使用者都有自己的風格。風格指南精靈必須讓捕捉個人風格變得容易。

- 在風格指南格式中加入 `sample_passages` 欄位；`profile/manager.go` 解析後傳遞給生成與審查提示
- 在精靈的風格步驟中，明確引導使用者：
  > 「貼上 2–3 段你喜歡的文字——你自己的作品，或你欣賞的作者的風格（請注意著作權）。這些範例會直接影響生成文字的節奏、用詞與句式結構。」
- 顯示範例將如何被使用的預覽，讓使用者了解其影響
- 在生成與修稿時，以具體 few-shot 範例注入範例段落，而非抽象的風格規則

### 故事結構模板

新手作者往往不知道從哪裡開始組織故事。結構模板降低了入門門檻。

- 在精靈中提供內建結構模板：三幕式、英雄旅程、雪花法與拯救貓咪法
- 每個模板預填章節數量、概要提示與關鍵轉折點標籤
- 使用者可從模板出發自由修改，或完全不使用模板
- 模板不鎖定故事——它們只是起步的鷹架

為什麼這很重要：

- 抽象的風格規則（「短句、冷靜語氣」）產出不一致；具體範例讓 LLM 能直接模仿
- 這是使用者能提供以改善本地模型生成品質的最高槓桿輸入
- 整個審查與生成流程都依賴這些資產檔案的存在與格式正確
- 要求非技術使用者手寫 `# 角色：小明\n- 個性：...` 是一道沉默的勸退牆
- 這是「安裝好工具」到「得到有用輸出」之間最重要的上線步驟
- 沒有結構引導，新手作者面對的是一張白紙；模板把「我不知道從哪裡開始」變成「我來調整這個大綱」

## Priority 3：寫作結構

目標：從章節審查升級到完整稿件組織。

- 加入 `章節 -> 場景` 層次，取代單純的章節檔案
- 建立包含概要、視角、衝突與目的欄位的場景看板
- 支援拖拉排序章節與場景
- 顯示每個場景的狀態，例如已審查、已改寫、未解決或需要世界觀補充
- 讓匯出功能能從選定的場景組裝完整稿件

為什麼這很重要：

- `novelWriter` 與 `Manuskript` 的強大之處在於幫助作者塑造結構，而不只是檢視文字
- 這是 Novel Assistant 與成熟小說寫作工具之間最明顯的落差

## Priority 2：故事 RAG 控制

目標：讓檢索更可預期、可調整、可信賴。

- 加入角色、世界觀、風格與章節上下文的個別來源切換
- 加入 `top-k` 與最低相似度門檻等檢索控制
- 加入行為審查、對白審查、修稿與世界觀衝突檢查等任務專屬檢索預設
- 顯示預期但未檢索到的上下文，提示可能遺漏的來源
- 加入來源權重或優先順序，讓關鍵正典檔案更難被忽略

### 術語表與世界詞典

目前的字串比對依賴角色名字明確出現。結構化術語表將這個能力延伸到故事世界中所有的專有名詞。

- 讓使用者定義術語表條目：專有名詞、地名、陣營名稱、創作術語及其定義
- 審查章節時自動偵測術語表中的詞彙，標記拼寫、含義或使用方式的不一致
- 將術語表條目連結到世界觀檔案，讓 RAG 流程在取得角色上下文的同時也能取得定義
- 審查章節時顯示術語表側欄，讓作者能隨時確認用詞
- 偵測章節中出現術語表詞彙、但對應的世界觀檔案未被檢索的情況，並作為遺漏上下文警告呈現

為什麼這很重要：

- 世界觀 Markdown 檔案是自由格式；沒有結構化索引記錄故事中存在哪些專有名詞
- 世界詞典讓故事宇宙可被查詢——LLM 能回答「這個術語是什麼？」而不是猜測
- 跨章節的術語一致性是長篇小說中最常見的編輯盲點之一

### 人稱代詞解析與角色偵測精確度（Issue #19）

目前的字串比對只有在角色名字明確出現時才能找到角色。完全用代詞書寫的場景會被靜默跳過，使一致性檢查不完整。

**Phase 1 — 提示層引導（低成本）**
- 在 `CheckBehaviorStream` 加入指令，告知 LLM 根據上下文將代詞（他 / 她 / he / she）視為可能指向目標角色
- 在 llama3.2 的 128K context window 範圍內，涵蓋大多數單章審查場景

**Phase 2 — 長文本滑動視窗**
- 將超過 context window 的章節切割成重疊區塊
- 在每個區塊注入角色摘要標頭，讓 LLM 具備跨區塊角色上下文
- 合併並去重跨區塊的發現

為什麼這很重要：

- 若無代詞解析，整個場景只用代詞指稱角色時，一致性檢查會產生零筆發現，讓結果變得誤導而非只是不完整

## Priority 4：雙語支援（中文 + 英文）

目標：讓英文寫作的使用者不需要中文格式的資產檔案就能使用工具。

- 支援英文格式的角色設定，與現有中文格式並存：
  ```
  # Character: John
  - Personality: ...
  - Core fear: ...
  - Speech style: ...
  ```
- 支援英文格式的風格指南與世界觀檔案
- 偵測章節語言並自動切換 LLM 提示語言（中文章節 → 中文提示，英文章節 → 英文提示）
- 將 UI 標籤與錯誤訊息翻譯成英文；在設定頁加入語言切換

為什麼這很重要：

- 目前 `profile/manager.go` 寫死了中文欄位標頭；英文資產檔案會產生空的角色資料與完全錯誤的審查輸出
- 雙語支援是讓專案能吸引非中文社群的最低可行步驟

## Priority 5：章節版本控制

目標：讓作者對每次編輯或改寫的章節都有安全、可還原的歷史記錄。

- 追蹤每個章節檔案的每次儲存版本，帶有時間戳與來源標籤（手動存檔、改寫、回填）
- 讓使用者能並排比較任意兩個版本的差異
- 支援一鍵還原到任意舊版本
- 可選擇為版本加上標籤（例如「結構改寫前」、「Beta 讀者稿」）

為什麼這很重要：

- 目前改寫章節並儲存會覆蓋舊版本且無法復原
- 正在修改長篇稿件的作者承受不起失去作品的風險；版本控制是任何認真的寫作工具都必須提供的基本安全保障

## Priority 6：自訂檢查類型

目標：讓使用者在四種內建類型之外，定義自己的審查維度。

- 讓使用者在設定中加入自訂檢查定義：名稱、提示模板與可選的檢索預設
- 自訂檢查會與行為 / 對白 / 世界觀 / 風格一起出現在審查頁面
- 內建檢查保持不變；自訂檢查為加法
- 支援匯出與匯入檢查定義，以便在專案或使用者之間共享

為什麼這很重要：

- 不同類型的小說有不同的一致性顧慮；硬科幻作者需要物理一致性檢查，歷史小說作者需要時代錯誤檢查
- 目前的四種檢查類型適用於一般小說，無法涵蓋所有作者的需求

## Priority 7：AI 輔助小說生成

目標：讓使用者透過相同的本地優先設定，從故事概念生成完整草稿。

這是最大的單一功能擴展。它將 Novel Assistant 從審查工作站升級為既能生成也能審查的完整寫作夥伴。

### Phase 1 — 大綱生成

- 使用者提供角色設定、世界觀筆記、風格指南與簡短故事前提
- LLM 生成結構化大綱：章節清單、每章概要、關鍵場景節點與字數目標
- 大綱在生成開始前可編輯；每章可獨立核准、調整或重新生成

### Phase 2 — 逐章生成

- 每章依序生成，使用：
  - 核准的章節概要與場景節點
  - 先前已生成章節的滾動摘要（壓縮以符合 context）
  - 透過 RAG 取得的完整角色與世界觀設定
  - 選定的風格指南
- 輸出透過 SSE 即時串流，與現有的審查與修稿流程相同
- 每章生成後，使用者可在繼續前審查、編輯或重新生成

### Phase 3 — 一致性迴圈

- 生成完成後，現有的行為、對白、世界觀與風格審查流程會自動對每章執行
- 發現以後置生成報告呈現，而非阻擋生成
- 使用者可在最終匯出前對標記的場景觸發定向修稿

### Phase 4 — 行內段落生成建議

- 在章節任意位置放置游標，觸發「從這裡繼續」以取得 AI 生成的下一段句子或段落
- 建議是行內插入，而非整章替換——作者逐一接受、拒絕或編輯
- 使用與完整生成相同的角色、世界觀與風格上下文，確保建議符合故事世界
- 解決「場景寫到一半卡住」的問題，不需要離開編輯器

### Phase 5 — 腦力激盪模式

- 作者提出開放性問題：「接下來可能發生什麼？」、「這個角色遇到……會怎麼做？」、「有哪三種方式可以提升張力？」
- LLM 以結構化選項回應（而非散文），每個選項附一句理由
- 作者挑選、捨棄或混合選項後再動筆——腦力激盪從不替作者寫章節
- 輸出可儲存為場景筆記供日後參考

為什麼這很重要：

- 多數本地 AI 寫作工具止步於聊天或單場景生成；帶有跨章節記憶的逐章生成是缺失的一塊
- 將生成與審查整合在同一工具中，讓作者永遠不需要在兩個 app 之間複製貼上以檢查一致性
- 行內建議解決了寫作中摩擦最高的時刻：場景寫到一半卡住、不知道下一步
- 腦力激盪模式呼應 Sudowrite 最常被使用的功能——降低空白頁恐懼，同時不取代作者的聲音

### 架構說明

- 新增 `internal/generation` 套件，處理大綱狀態、章節排序與滾動摘要管理
- 新增 `generation.html` 頁面，帶有逐步精靈 UI：設定 → 大綱 → 生成 → 審查 → 匯出
- 與現有的 `internal/profile`、`internal/embedder`、`internal/checker` 與 `internal/vectorstore` 共用
- 大綱狀態持久化於 `data/generation/`，讓中斷的工作階段可以恢復

### 相依性

- 需要先完成 Priority 1（場景層次）；生成以場景為目標，而非原始章節檔案
- 在 Priority 2（故事 RAG 控制）穩定後效果最佳，確保生成時的檢索可靠

為什麼這很重要：

- 多數本地 AI 寫作工具止步於聊天或單場景生成；帶有跨章節記憶的逐章生成是缺失的一塊
- 將生成與審查整合在同一工具中，讓作者永遠不需要在兩個 app 之間複製貼上以檢查一致性

## Priority 8：稿件建置與交付

目標：以更接近真實作者工作流程的格式匯出寫作專案。

- 跨章節與場景的完整稿件匯出
- 匯出至多種格式，例如 `Markdown`、`HTML`、`DOCX` 與 `PDF`
- 提供可選的附錄，包括審查筆記、修稿摘要與追蹤器狀態
- 支援依草稿狀態或場景標籤篩選的章節組合
- 加入可分享的審查包，供編輯或 Beta 讀者使用

為什麼這很重要：

- 成熟的寫作工具不只停在編輯；它們幫助你產出真正要寄出去的稿件

## Priority 9：多專案安全性

目標：讓 app 在長期重度使用下保持安全。

- 加入專案 / 工作區隔離，取代單一 `data/` 根目錄
- 加入每個專案獨立的設定與向量儲存
- 加入專案匯入 / 匯出與遷移工具
- 加入更安全的還原流程，在覆蓋前先預覽
- 加入自動快照輪替與還原歷史

為什麼這很重要：

- 一旦作者將數個月的故事作品託付給工具，專案安全性就跟模型品質一樣重要

## Priority 10：審查品質回饋（僅限本地）

目標：讓使用者標記哪些審查發現有用，純粹用於改善自己的本地體驗——資料永遠不會離開裝置。

- 在歷史頁面的個別審查發現加入讚 / 踩按鈕
- 將接受 / 拒絕標籤在本地與現有的 `reviewhistory` 項目一起儲存
- 使用標籤來呈現每種檢查類型的個人接受率摘要
- 不匯出、不選擇性分享、不遙測——標籤留在使用者的機器上

為什麼這很重要：

- 本專案的核心承諾是稿件永遠不會離開使用者的電腦；任何資料蒐集機制，即使是選擇性的，都有侵蝕信任的風險
- 本地回饋標籤仍有價值：使用者可以看到哪些檢查類型對自己的寫作持續產生有用的發現，並據此調整預設設定

## Priority 11：成熟的應用程式外表

目標：降低採用與維護的摩擦。

- 加入認證模式或可選的本地存取保護
- 加入更豐富的截圖與上線文件
- 加入 `go run` 與 Docker 以外的打包 / 安裝路徑
- 加入索引大小、檢索品質與模型延遲等指標或除錯視圖
- 加入 Windows、macOS、Linux 與容器化設定的相容性說明

## Priority 12：寫作體驗

目標：讓 Novel Assistant 從審查工具升級為作者真正想在裡面寫作的地方。

### 寫作統計與每日目標

- 追蹤每章與每次工作階段的字數，並附時間戳
- 讓使用者設定每日或每週字數目標
- 在主控台顯示寫作連續天數、進度條與累計字數圖表
- 標出最近沒有動靜的章節，讓作者知道自己停在哪裡

### 無干擾寫作模式

- 全螢幕編輯器，隱藏側欄、最簡 chrome，並對當前段落做柔焦高亮
- 單一鍵盤快捷鍵切換
- 保留所有審查與修稿工具——它們可從摺疊選單取用，不是被移除
- 可選打字機滾動，讓目前行保持置中

為什麼這很重要：

- 字數目標是驅動每日寫作習慣最直接的槓桿——看到連續天數的作者不想打破它
- novelWriter 與 Manuskript 都提供無干擾模式；使用者將它列為留下來的主要理由
- 目前的 UI 是工具導向；無干擾模式讓它變成寫作導向
- 當工具是寫作發生的地方而不只是審查的地方，留存率更高

## 建議開發順序

若開發持續按序進行，以下是建議順序：

1. 首次體驗（Issues #20–24）
2. 寫作統計與每日目標——成本低，留存影響高
3. 引導式故事世界設定（角色、世界觀、風格的精靈 UI，含結構模板）
4. 術語表與世界詞典
5. 場景層次與場景看板
6. 無干擾寫作模式
7. 代詞解析與檢索精確度
8. 雙語支援（中文 + 英文）
9. 章節版本控制
10. 自訂檢查類型
11. AI 輔助小說生成（大綱 → 逐章 → 行內建議 → 腦力激盪）
12. 完整稿件匯出
13. 多專案工作區支援
14. 審查回饋蒐集（在有足夠使用者基礎之後）
15. 可選本地認證與更強的維運工具
